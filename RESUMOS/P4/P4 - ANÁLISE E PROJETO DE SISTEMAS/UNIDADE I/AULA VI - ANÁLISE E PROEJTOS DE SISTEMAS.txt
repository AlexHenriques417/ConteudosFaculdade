AULA VI - ANÁLISE E PROEJTOS DE SISTEMAS - UNIDADE I

- INTRODUÇÃO
O DIAGRAMA DE CASOS DE USO MOSTRA O COMPORTAMENTO EXTERNO DO SISTEMA
É O MAIS ABSTRATO, FLEXÍVEL E INFORMAL DA UML
UTILIZADO NAS ETAPAS DE LEVANTAMENTO E ANÁLISE DE REQUISITOS
MODIFICADO DURANTE TODO O PROCESSO DE ENGENHARIA
SERVE DE BASE PARA OUTROS DIAGRAMAS
PODE SER USADO COMO PROTÓTIPO PARA CLIENTES
FACILITA A ORGANIZAÇÃO DOS REQUISITOS
MOSTRA O SISTEMA PELA PERSPECTIVA DO USUÁRIO
OBJETIVO: VISÃO EXTERNA DAS FUNÇÕES E SERVIÇOS QUE O SISTEMA DEVE OFERECER
DESCREVE O QUE O SISTEMA FAZ, MAS NÃO COMO

- QUANDO UTILIZAR
FERRAMENTA PARA ENTENDER REQUISITOS FUNCIONAIS
PRIMEIRA VERSÃO LOGO NO INÍCIO DO DESENVOLVIMENTO
DETALHES DEVEM SER ADICIONADOS POSTERIORMENTE
NÃO DEVE SER TORNADO EXCESSIVAMENTE COMPLEXO

- BENEFÍCIOS
FOCO NO USUÁRIO DO SISTEMA
FACILIDADE DE ENTENDIMENTO
AJUDA NO ACORDO ENTRE STAKEHOLDERS

- CASO DE USO
REQUISITO FUNCIONAL DO SISTEMA
SEQUÊNCIA DE INTERAÇÕES ENTRE SISTEMA E ATORES
DESCRIÇÃO TEXTUAL DAS FUNCIONALIDADES (CENÁRIOS)
ESPECIFICA VÁRIOS CENÁRIOS (EX.: RETIRAR DINHEIRO)
ASSOCIADO A UMA OU MAIS TELAS
REPRESENTADO POR ELIPSES (VERBO + OBJETO)
EXEMPLOS: “CADASTRAR CLIENTES”, “SACAR DINHEIRO”, “VERIFICAR PEDIDO”

- DIAGRAMA DE CASOS DE USO
FACILITA ENTENDIMENTO MOSTRANDO VISÃO EXTERNA.
MODELA O CONTEXTO DE UM SISTEMA OU SUBSISTEMA.
DELIMITA O ESCOPO E AS FUNCIONALIDADES.

- ELEMENTOS DO DIAGRAMA
ATORES → PAPÉIS DE ELEMENTOS EXTERNOS (PESSOAS, HARDWARE, OUTROS SISTEMAS)
CASOS DE USO → FUNCIONALIDADES DO SISTEMA
RELACIONAMENTOS:
ASSOCIAÇÃO
GENERALIZAÇÃO
EXTENSÃO (<<EXTENDS>>)
INCLUSÃO (<<INCLUDE>>)
FRONTEIRA DO SISTEMA → DELIMITA A ÁREA DE ATUAÇÃO.

- RELACIONAMENTOS
ASSOCIAÇÃO: LIGA ATORES A CASOS DE USO
GENERALIZAÇÃO: HERANÇA ENTRE ATORES OU CASOS DE USO
EXTENSÃO: COMPORTAMENTOS OPCIONAIS, EXECUTADOS EM CERTAS SITUAÇÕES
INCLUSÃO: COMPORTAMENTOS OBRIGATÓRIOS, USADOS EM MAIS DE UM CASO DE USO

- ERROS FREQUENTES
FAZER CASOS DE USO PEQUENOS DEMAIS (EX.: INCREMENTAR CONTADOR)
FAZER CASOS DE USO GRANDES DEMAIS (EX.: REALIZAR COMPRA)
UM CASO DE USO DEVE GERAR RESULTADO PERCEPTÍVEL PARA O ATOR

- BENEFÍCIOS DO MODELO DE CASOS DE USO
CLIENTE: CAPTURA E VALIDA REQUISITOS, AJUDA A DEFINIR ESCOPO, ESTIMAR PRAZOS E CUSTOS, BASE PARA TESTES DE HOMOLOGAÇÃO
USUÁRIO: DESCREVE A INTERAÇÃO DO USUÁRIO COM O SISTEMA
GERENTE DE PROJETOS: AUXILIA EM ESTIMATIVA DE CUSTOS, RISCOS, RASTREAMENTO E MAPEAMENTO DE REQUISITOS
ARQUITETO DO SISTEMA: IDENTIFICA REQUISITOS ARQUITETURAIS, AVALIA TECNOLOGIAS, GARANTE CONSISTÊNCIA E COERÊNCIA
DESENVOLVEDOR: USA OS MODELOS COMO BASE PARA CODIFICAÇÃO
MANTENEDOR: GUIA PARA MODIFICAÇÕES E EVOLUÇÃO DA ARQUITETURA

- ESPECIFICAÇÃO DE CASOS DE USO
A UML NÃO DEFINE PADRÃO FIXO PARA A DESCRIÇÃO TEXTUAL
A DESCRIÇÃO É MAIS IMPORTANTE QUE O DIAGRAMA
ESTILOS DE DESCRIÇÃO:
INFORMAL: NOME E BREVE TEXTO DA FUNCIONALIDADE
TÍPICA: ATORES, PRÉ-REQUISITOS, FLUXO NORMAL, FLUXOS ALTERNATIVOS
DETALHADA: IDENTIFICADOR, NOME, RESUMO, ATORES, PRÉ-COND. E PÓS-COND., FLUXOS PRINCIPAL, ALTERNATIVOS E DE EXCEÇÃO, RESTRIÇÕES

- ELEMENTOS DA DESCRIÇÃO DETALHADA
IDENTIFICADOR: CÓDIGO ÚNICO (EX.: CSU01, CSU02)
NOME: DEVE CORRESPONDER AO DIAGRAMA
DESCRIÇÃO/RESUMO: OBJETIVO DO ATO
ATOR PRIMÁRIO: INICIA O CASO DE USO (UM ÚNICO)
ATORES SECUNDÁRIOS: APOIAM A INTERAÇÃO (ZERO OU MAIS)
PRÉ-COND: CONDIÇÕES NECESSÁRIAS PARA INICIAR O CASO
PÓS-COND: CONDIÇÕES QUE DEVEM ESTAR PRESENTES AO FINAL
FLUXO PRINCIPAL: SEQUÊNCIA NORMAL DE PASSOS, DO PONTO DE VISTA DO USUÁRIO
FLUXOS ALTERNATIVOS: OPÇÕES OU VARIAÇÕES DE USO
FLUXOS DE EXCEÇÃO: SITUAÇÕES QUE COMPROMETEM O FUNCIONAMENTO

- FORMATOS DE DESCRIÇÃO
CONTÍNUO: TEXTO LIVRE 
NUMERADO: PASSOS SEQUENCIAIS, MAIS ORGANIZADO

- RELACIONAMENTOS NA ESPECIFICAÇÃO
INCLUSÃO (<<INCLUDE>>): COMPORTAMENTO OBRIGATÓRIO, REPETIDO EM MAIS DE UM CASO DE USO
EXTENSÃO (<<EXTENDS>>): COMPORTAMENTO OPCIONAL, INSERIDO EM SITUAÇÕES ESPECÍFICAS
HERANÇA: CASO DE USO FILHO HERDA, SOBRESCREVE OU ACRESCENTA AO CASO DE USO PAI
