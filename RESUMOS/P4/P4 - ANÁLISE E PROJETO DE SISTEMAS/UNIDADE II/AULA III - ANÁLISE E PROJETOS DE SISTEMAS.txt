AULA III - ANÁLISE E PROJETOS DE SISTEMAS - UNIDADE II

- PERIGOS DO USO DE PADRÕES
O USO EXCESSIVO E DESCONTROLADO DE PADRÕES PODE GERAR PROJETOS SOBRECARREGADOS
DESENVOLVEDORES PRECISAM:
TEMPO PARA ENTENDER OS CATÁLOGOS DE PADRÕES
ACESSO FÁCIL AOS CATÁLOGOS
TREINAMENTO NO USO ADEQUADO DE PADRÕES

- ANTI-PADRÕES
PADRÕES INDICAM BOAS ESTRUTURAS DE ANÁLISE, PROJETO OU IMPLEMENTAÇÃO
ANTI-PADRÕES INDICAM O OPOSTO: ERROS COMUNS E FALHAS DE PROJETO
FOCO: DETECTAR E CORRIGIR PRÁTICAS RUINS NO DESENVOLVIMENTO DE SOFTWARE

- ELEMENTOS ESSENCIAIS DE UM PADRÃO DE SOFTWARE
NOME: RESUME O PROBLEMA, A SOLUÇÃO E AS CONSEQUÊNCIAS; DEVE SER SUGESTIVO E FÁCIL DE LEMBRAR
PROBLEMA: DESCREVE QUANDO E ONDE APLICAR O PADRÃO; CONTEXTOS, SINTOMAS E CONDIÇÕES
SOLUÇÃO: DETALHA ELEMENTOS, RELACIONAMENTOS, RESPONSABILIDADES E COLABORADORES
CONSEQUÊNCIAS: RESULTADOS, IMPACTOS, VANTAGENS E DESVANTAGENS DO PADRÃO
OUTROS ELEMENTOS: EXEMPLOS, PADRÕES RELACIONADOS, NOMES ALTERNATIVOS E AMOSTRAS DE CÓDIGO

- BENEFÍCIOS DO USO DE NOMES PADRONIZADOS
FACILITAM A COMUNICAÇÃO ENTRE DESENVOLVEDORES
PERMITEM DISCUSSÕES EM ALTO NÍVEL DE ABSTRAÇÃO
SIMPLIFICAM A DOCUMENTAÇÃO
EXEMPLO: PADRÃO “SINGLETON” DEFINE UMA ÚNICA INSTÂNCIA DE UMA CLASSE

- PADRÕES GRASP
SIGNIFICA GENERAL RESPONSIBILITY ASSIGNMENT SOFTWARE PATTERNS
DESENVOLVIDOS POR CRAIG LARMAN
DEFINEM UMA FORMA SISTEMÁTICA DE ATRIBUIR RESPONSABILIDADES ÀS CLASSES
RESPONSABILIDADES PODEM SER DE DOIS TIPOS:
DE CONHECIMENTO (KNOWING): DADOS, RELAÇÕES E CÁLCULOS
DE REALIZAÇÃO (DOING): AÇÕES, CONTROLE E COORDENAÇÃO DE OUTROS OBJETOS

- PRINCIPAIS PADRÕES GRASP
INFORMATION EXPERT
CREATOR
HIGH COHESION
LOW COUPLING
CONTROLLER

- PADRÃO GRASP: CREATOR
PROBLEMA: QUEM DEVE SER RESPONSÁVEL POR CRIAR INSTÂNCIAS DE OUTRAS CLASSES?
SOLUÇÃO: ATRIBUIR ESSA RESPONSABILIDADE À CLASSE QUE:
AGREGA OU CONTÉM OS OBJETOS
USA OS OBJETOS
POSSUI OS DADOS INICIAIS PARA A CRIAÇÃO
TEM CONHECIMENTO SOBRE QUAL SUBCLASSE CRIAR
EXEMPLO:
    A CLASSE CARRINHODECOMPRAS CRIA E GERENCIA ITEMDECOMPRA
    A CLASSE PRODUTO CRIA SUAS PRÓPRIAS INSTÂNCIAS

- PADRÃO GRASP: LOW COUPLING (BAIXO ACOPLAMENTO)
PROBLEMA: COMO MINIMIZAR DEPENDÊNCIAS ENTRE CLASSES E MAXIMIZAR O REUSO?
SOLUÇÃO: DISTRIBUIR RESPONSABILIDADES DE FORMA A REDUZIR LIGAÇÕES ENTRE CLASSES
BENEFÍCIOS:
    FACILITA A MANUTENÇÃO E A COMPREENSÃO DO SISTEMA
    PERMITE TROCAR IMPLEMENTAÇÕES SEM IMPACTAR OUTRAS PARTES
EXEMPLO: USO DE INTERFACES COMO “FORMAGEOMETRICA”, COM MÉTODO ABSTRATO “CALCULAR_AREA()”, PARA PERMITIR TROCA DE IMPLEMENTAÇÕES

- PADRÃO GRASP: HIGH COHESION (ALTA COESÃO)
PROBLEMA: EVITAR CLASSES QUE FAZEM MUITAS TAREFAS DIFERENTES
SOLUÇÃO: ORGANIZAR AS RESPONSABILIDADES PARA QUE CADA CLASSE TENHA UM PROPÓSITO CLARO
BENEFÍCIOS:
    CLASSES FÁCEIS DE ENTENDER, REUTILIZAR E MANTER
    MENOS AFETADAS POR MUDANÇAS
EXEMPLO: A CLASSE CALCULADORA REALIZA APENAS OPERAÇÕES MATEMÁTICAS, COM MÉTODOS RELACIONADOS ENTRE SI

- PADRÃO GRASP: CONTROLLER
PROBLEMA: QUEM DEVE RECEBER E COORDENAR AS SOLICITAÇÕES DE USUÁRIOS?
SOLUÇÃO: CRIAR UMA CLASSE “CONTROLADOR” PARA GERENCIAR INTERAÇÕES ENTRE A INTERFACE DO USUÁRIO E O SISTEMA
FUNÇÃO: RECEBER SOLICITAÇÕES, ACIONAR AÇÕES INTERNAS E RETORNAR RESULTADOS
EXEMPLO:
    PEDIDOCONTROLLER RECEBE PEDIDOS, GERENCIA ITENS DO CARRINHO, CALCULA O TOTAL E RETORNA A CONFIRMAÇÃO AO CLIENTE

- CONCLUSÃO
OS PADRÕES GRASP AJUDAM A DISTRIBUIR RESPONSABILIDADES DE MANEIRA CLARA
PROMOVEM BAIXO ACOPLAMENTO, ALTA COESÃO E MELHOR ORGANIZAÇÃO DO CÓDIGO
SÃO FUNDAMENTAIS PARA PROJETOS ORIENTADOS A OBJETOS ROBUSTOS, FLEXÍVEIS E REUTILIZÁVEIS
