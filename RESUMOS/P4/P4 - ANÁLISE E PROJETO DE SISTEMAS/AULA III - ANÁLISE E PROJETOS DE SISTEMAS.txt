AULA III - ANÁLISE E PROJETOS DE SISTEMAS

- LEVANTAMENTO E ANÁLISE DE REQUISITOS
UTILIZAR TODAS AS INFORMAÇÕES DISPONIVEIS PARA IDENTIFICAR AS FONTES DE REQUISITOS 
IDENTIFICAR AS FUNÇÕES QUE O SISTEMA DEVERÁ DISPONIBILIZAR

- LEVANTAMENTO DE REQUISITOS
BUSCAR TODAS AS INFORMAÇÕES POSSÍVEIS
FUNÇÕES QUE O SISTEMA DEVE EXECUTAR E AS RESTRIÇÕES SOBRE AS QUAIS O SISTEMA DEVE OPERAR
O PROUTO DESSA ETAPA SERÁ O DOCUMENTO DE REQUISITOS

- ANÁLISE DE REQUISITOS
ESTRUTURAR E DETALHAR OS REQUISITOS 
DE FORMA QUE ELES POSSAM SER ABORDADOS NA FASE DE MODELAGEM DO SISTEMA
ANALISADOS E CONFIRMADOS PELOS PRINCIPAIS INTERESSADOS DO PROJETO

- COMPREENSÃO DO DOMÍNIO
OS ANALISTAS DEVEM DESENVOLVER SUA COMPREENSÃO DO DOMÍNIO DA APLICAÇÃO

- COLETA DE REQUISITOS
INTERAGIR COM OS STAKHOLDERS DO SISTEMA PARA DESCOBRIR SEUS REQUISITOS

- CLASSIFICAÇÃO 
ORGANIZAR EM GRUPOS COERENTES

- RESOLUÇÃO DE CONFLITOS
SOLUCIONAR CONFLITOS

- DEFINIÇÃO DAS PRIORIDADES
SEMPRE TERÃO COISAS MAIS IMPORTANTES QUE OUTROS

- VERIFICAÇÃO DE REQUISITOS
SÃO VERIFICADOS SE ESTÃO COMPLETOS E CONSISTENTES

- ENGENHARIA DE REQUISITOS
PROCESSO DE DESCOBRIR, ANALISAR, DOCUMENTAR E VERIFICAR REQUISITOS

- REQUISITOS DE USUÁRIO
LINGUAGEM NATURAL COM DIAGRAMAS
SERVIÇOS QUE O SISTEMA DEVE FAVORECR
RESTRIÇÕES COM AS QUAIS ESTE DEVE OPERAR

- REQUISITOS DE SISTEMA
DESCRIÇÕES MAIS DETALHADA DAS FUNÇÕES
SERVIÇOS E RESTRIÇÕES OPERACIONAIS DO SISTEMA DE SOFTWARE
DEVEM SER PADRONIZADAS, COMPLETOS E CONSISTENTES
USADAS PELA EQUIPE DE DESENVOLVIMENTO

- REQUISITOS DE SOFTWARE

- REQUISITOS FUNCIONAIS
DECLARAções DE SERVIÇO QUE O SISTEMA DEVE FORNECER, COMO AS ENTRADAS E COMO SE COMPORTAR

- REQUISITOS NÃO FUNCIONAIS 
SÃO RESTRIÇÕES AOS SERVIÇOS OFERECIDOS PELO SISTEMA
ATRIBUTOS OU QUALIDADES DO SISTEMA QUE NÃO SÃO REPRESENTADO ATRAVES DE FUNÇÕES
DESCREVEM CARACTERISTICAS 

- USABILIDADE 
O QUAL FÁCIL DE USAR

- CONFIBALIDADE 
GRAU DE CONFIANÇA

- DESEMPENHO
TEMPO DE RESPOSTA PERFORMANCE

- MANUTENIBILIDADE
EVOLUÇÃO

- ESCABILIDADE
CRESCER E DIMINUIR DE ACORDO COM A DEMANDA

- PORTABILIDADE
VÁRIAS PLATAFORMAS

- REQUISITOS DE USUÁRIO 
DESCRITOS DE FORMA ABSTRATA PARA SEREM COMPREENDIDOS PELOS USUÁRIOS

- REQUISITOS DE SISTEMA
DESCREVEM EM DETALHES AS FUNÇÕES DO SISTEMA SUAS ENTRADAS E SAÍDAS EXCEÇÕES

- TÉCNICAS PARA COLETA DE REQUISITOS
ENTREVISTAS BRAINSTORMING, PROTOTIPAÇÃO, ETNOGRAFIA, QUESTIONÁRIOS E CASOS DE USO

- TIPOS DE ENTREVISTA 

- ENTREVISTA FECHADAS ( ESTRUTURADAS )
UM CONJUNTO DE QUESTÕES PREDEFINIDAS SÃO RESPONDIDAS

- ENTREVISTAS ABERTAS ( NÃO ESTRUTURADAS )
CONVERSA INFORMAL, UMA VARIEDADE DE ASSUNTOS SÃO EXPLORADOS COM OS STAKEHOLDERS

- SEMI-ESTRUTURADAS 
QUESTÕES ABERTAS E OUTRAS OBJETIVAS

- BRAINSTORMING 
TEMPESTADE DE IDEIAS VISA FACILITAR A PRODUÇÃO DE SOLUÇÕES ORIGINAIS E POSSUI DUAS FASES PRINICPAIS :

- PRODUÇÃO DE IDEIAS
- AVALIAÇÃO DAS IDEIAS PROPOSTAS

- OBJETIVO PRINCIPAL
PRODUZIR UM MAIOR NÚMERO DE IDEIAS POSSÍVEIS

- DOCUMENTO DE REQUISITO
INTRODUÇÃO E VISÃO GERAL DO DOCUMENTO
DESCRIÇÃO DE REQUISITOS FUNCIONAIS
DESCRIÇÃO DE REQUISITOS NÃO-FUNCIONAIS
ESCOPO NÃO CONTEMPLADO ( DE FUNCIONALIDADES )
DOCUMENTAÇÃO DE APOIO
